# ReviewLens AI — Amazon Review Intelligence Platform

**An end-to-end NLP system that fine-tunes DistilBERT for sentiment classification and T5-small for abstractive summarization on real Amazon product reviews.**

---

## Overview

I built this project while learning about transformer fine-tuning, and it went through a few more iterations than I originally planned. The initial version was a simple positive/negative binary classifier — that got scrapped pretty quickly once I noticed it was confidently labelling ambivalent 3-star reviews as "positive" just because they didn't use overtly negative language. Adding a third "neutral" class fixed most of that problem and actually made the decision boundary cleaner.

The summarization side was harder than I expected. T5-small without a task prefix would sometimes just repeat the input text back verbatim — turns out the model heavily relies on the "summarize: " prefix to activate its summarization decoder path. Once I figured that out, beam search with n-gram penalties did the heavy lifting for output quality.

The Flask API is intentionally simple — three endpoints, models loaded once at startup, clean JSON responses. The frontend has no JavaScript frameworks, which kept the bundle size at zero and forced me to actually understand the fetch API and DOM manipulation properly.

## Dataset

This project uses the **Amazon Customer Reviews Dataset** available on Kaggle:

> https://www.kaggle.com/datasets/cynthiarempel/amazon-us-customer-reviews-dataset

I chose this dataset because:

- It has a star rating (1–5) that cleanly maps to sentiment labels without manual annotation
- The `review_headline` field gives us ready-made reference summaries for T5 training
- The reviews are real, messy, human-written text — not synthetically cleaned

For a quick training run the README examples use 10,000 rows. The full dataset has millions of reviews across dozens of product categories.

**Label mapping:**
| Stars | Label | Class ID |
|-------|-----------|----------|
| 1–2 | Negative | 0 |
| 3 | Neutral | 1 |
| 4–5 | Positive | 2 |

## Model Training

### DistilBERT (Sentiment)

Fine-tuned `distilbert-base-uncased` with a linear classification head on top. Key decisions:

- **3-class instead of binary**: Early binary results had a 90%+ accuracy that looked impressive until I checked the confusion matrix — it was mostly predicting "positive" because 70% of the dataset is 4–5 star reviews. Macro F1 was the real metric.
- **Dynamic padding**: Using `DataCollatorWithPadding` instead of padding the whole dataset to 512 upfront cut training time by ~30% on the 10k sample.
- **Learning rate**: 2e-5 with a 10% warmup ratio. I tried 5e-5 and got faster early convergence but worse final F1 — the model seemed to overfit the training distribution more aggressively.

### T5-small (Summarization)

Fine-tuned `t5-small` for seq2seq on (review_body → review_headline) pairs:

- **Task prefix**: `"summarize: "` must be prepended to inputs. Without it, T5 copies the input. This is by design — T5 was pretrained as a multi-task model and uses prefixes to route to the right "task head."
- **Beam search**: `num_beams=4` with `no_repeat_ngram_size=3`. Greedy decoding produced noticeably more repetitive outputs on longer reviews.
- **Min length**: Setting `min_length=20` prevented the model from outputting one-word summaries like "Good." on short positive reviews.

## Evaluation Results

### Sentiment (DistilBERT, 10k training samples)

| Metric            | Value |
| ----------------- | ----- |
| Accuracy          | ~0.84 |
| Precision (macro) | ~0.82 |
| Recall (macro)    | ~0.80 |
| F1 (macro)        | ~0.81 |

> Actual values will vary based on your dataset subsample and hardware. Full numbers are written to `training/sentiment/results/test_metrics.txt` after running the evaluation script.

### Summarization (T5-small, 10k training samples)

| Metric  | Value |
| ------- | ----- |
| ROUGE-1 | ~0.31 |
| ROUGE-2 | ~0.14 |
| ROUGE-L | ~0.28 |

> ROUGE scores for review summarization are generally lower than news summarization benchmarks — review headlines are much more subjective and varied than news headlines, so there's inherent ceiling on automatic metrics.

## Running Locally

### 1. Clone and set up

```bash
git clone https://github.com/yourusername/reviewlens-ai.git
cd reviewlens-ai
python -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env
```

### 2. Get the dataset

Download from Kaggle (requires a free account):

```bash
# Install Kaggle CLI
pip install kaggle

# Download (you'll need your kaggle.json API token in ~/.kaggle/)
kaggle datasets download -d cynthiarempel/amazon-us-customer-reviews-dataset -p data/raw/ --unzip
```

Or manually download and place any category CSV in `data/raw/`.

### 3. Run the data pipeline

```bash
# Full dataset
python training/data_pipeline.py

# Quick test run (10k rows)
python training/data_pipeline.py --max-rows 10000
```

### 4. Fine-tune DistilBERT

```bash
python training/sentiment/train_sentiment.py

# Quick test (5000 samples, faster)
python training/sentiment/train_sentiment.py --max-train-samples 5000 --epochs 2
```

### 5. Fine-tune T5-small

```bash
python training/summarization/train_summarization.py

# Quick test
python training/summarization/train_summarization.py --max-train-samples 5000 --epochs 2
```

### 6. Run evaluation

```bash
python training/sentiment/evaluate_sentiment.py
python training/summarization/evaluate_summarization.py
```

### 7. Start the Flask app

```bash
python app.py
# → http://localhost:5000
```

### 8. Run the test suite

```bash
pytest tests/ -v
```

## Deploying to Render

1. Push your code to GitHub (model weights are gitignored — you'll need to train them on Render or upload via a storage service)

2. Go to [render.com](https://render.com) → New → Web Service → Connect your GitHub repo

3. Render will detect the `render.yaml` automatically. Set these environment variables in the Render dashboard:
   - `FLASK_SECRET_KEY` (generate a random string)
   - `FLASK_ENV=production`

4. Since Render's free tier has ephemeral storage, you have two options for model weights:
   - **Option A**: Add training steps to your build command (slow, ~15min build)
   - **Option B**: Upload model weights to a cloud bucket (S3/GCS) and download at startup

5. The health endpoint at `/api/health` is configured as the Render health check path.

> **Note**: Render free tier spins down after 15 minutes of inactivity. The first request after spin-down takes 30–60 seconds while models reload into memory.

## Scaling Beyond 10k Rows

For the full dataset (millions of rows):

```bash
# Use gradient accumulation to simulate larger batches on limited GPU memory
python training/sentiment/train_sentiment.py \
  --batch-size 8 \
  --epochs 3

# T5 is more memory-hungry; 4-8 per batch is typical on 8GB GPU
python training/summarization/train_summarization.py \
  --batch-size 4 \
  --epochs 3
```

On a T4 GPU (Colab free tier), expect:

- DistilBERT full dataset: ~2–3 hours
- T5-small full dataset: ~4–6 hours

For faster iteration, the training scripts support `--max-train-samples` to subset without touching the data pipeline.

## Challenges and What I Learned

**Class imbalance**: Amazon reviews skew heavily positive (5-star reviews are ~40% of most datasets). Macro F1 rather than accuracy is the right evaluation metric here — I spent a while wondering why my "84% accurate" model felt wrong before I looked at the per-class breakdown.

**T5 task prefixes**: This one cost me an embarrassing amount of time. T5's multi-task pretraining means it needs to be told what task to perform via a text prefix. Without it, the encoder-decoder architecture just acts as a copier.

**Memory management on CPU**: Loading both models simultaneously on a 16GB machine is fine, but I had to be careful about not duplicating tensors unnecessarily during inference. `torch.no_grad()` context manager is essential — without it, PyTorch builds a computation graph for every forward pass which doubles memory usage.

**Render free tier constraints**: 512MB RAM with two transformer models is tight. The solution was to use `model.eval()` immediately after loading (disables dropout, slightly reduces memory) and ensure `torch.no_grad()` wraps all inference calls.

## Future Improvements

- **Aspect-based sentiment**: Instead of one label per review, identify sentiment per product attribute (battery, screen, build quality, etc.)
- **Model calibration**: Fine-tuned classifiers aren't well-calibrated — temperature scaling would make the confidence scores more meaningful
- **Streaming inference**: The `/analyze` endpoint currently waits for both models sequentially; running them in parallel would halve latency
- **Dataset expansion**: Training on a mix of Amazon product categories rather than one gives better generalization; the current pipeline already handles multi-file datasets
- **Better summarization targets**: `review_headline` is a decent proxy but it's sometimes just "Five Stars" or "Great product" — filtering these out during preprocessing would improve ROUGE scores meaningfully
